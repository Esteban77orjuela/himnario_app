# ADR-004: Estrategia de Testing

## Contexto
App React Native con lógica de dominio (transposición, parsing), servicios (scraper) y estado (Zustand).

## Decisión
**Jest 29 + jest-expo** con pruebas unitarias por capa.

### Stack
```
jest@29 + jest-expo@~54.0.0 + ts-jest
```

### Por qué Jest 29 (no 30+)
Expo 54 no es compatible con Jest 30+. Jest 29 es LTS estable.

### Pirámide de testing aplicada
```
    ╱╲
   ╱ E2E ╲      → No implementado (app 100% offline sin servidor)
  ╱────────╲
 ╱ Integration ╲ → Tests de store (Zustand) y flujos combinados
╱────────────────╲
╱   Unitarios     ╲ → Tests de utilidades y servicios (45 tests)
```

### Estrategia por archivo
| Capa | Archivo | Tipo | Tests |
|------|---------|------|-------|
| Utils | chordTransposer.ts | Unitario | 11 — transposición EN/ES, bemoles, sostenidos, sufijos |
| Utils | lyricsParser.ts | Unitario | 9 — parseo, detección, texto plano |
| Utils | extractChords.ts | Unitario | (pendiente) |
| Services | scraperService.ts | Unitario | 6 — HTML parsing, errores, limpieza |
| Data | chordDiagrams.ts | Unitario | 14 — lookup, auto-generación, invalid names |
| Store | useAppStore.ts | Unitario | (pendiente) — creación, actualización, persistencia |

### Mocking
- **AsyncStorage**: `@react-native-async-storage/async-storage/jest/async-storage-mock`
- **fetch**: `jest.fn()` con respuestas mockeadas en scraperService
- **Módulos nativos**: Excludes en `transformIgnorePatterns`

## Consecuencias
- No hay tests de componentes (React Testing Library) — prioridad baja para app sin servidor
- Coverage mínimo objetivo: 80% en utils/, 60% global
- Los tests del store requieren mock de AsyncStorage
